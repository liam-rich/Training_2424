# Test Data Organization

## Learning Objectives

By the end of this reading you will be able to:

- Describe the key **test data strategies**: synthetic, masked/anonymized, and subset production copies.
- Define **test data management (TDM)** and explain why it is a first-class concern in professional test organizations.
- Apply principles of **determinism, isolation, and privacy** to test environment data.
- Design a simple **baseline persona system** for a test environment.

---

## Why This Matters

"It worked in staging" is one of the most common — and frustrating — phrases in software delivery. Often, the root cause is not a code difference between staging and production, but a **data difference**. The staging database had missing records, stale state, or completely different data distributions from what real users produce.

Equally common: "The CI tests are failing randomly again." Most automated test flakiness traces to **shared mutable test data** — tests interfering with each other through a shared database, shared user accounts, or shared state that is never cleaned up.

Test data organization is the discipline that prevents both problems. It is the difference between a testing environment that developers and testers trust, and one that everyone suspects.

## The Concept

### The Three Data Strategies

**Strategy 1: Synthetic Data (Preferred for Most Cases)**

Synthetic data is **completely fabricated** — generated by code or tooling, not derived from real user records. It has no real people in it, making it safe to:
- Include in version-controlled repositories.
- Share across teams and contractors.
- Inspect in logs without privacy concerns.
- Use in CI environments without compliance risk.

**Tools for generating synthetic data:**

| Language | Tool | What It Generates |
|---------|------|-------------------|
| Python | Faker | Names, emails, addresses, dates, credit card numbers (fake), phone numbers, UUIDs |
| Java | JavaFaker | Same categories as Python Faker |
| JavaScript | Faker.js | Same categories |
| Any | Factory Boy (Python) | Object factories for model-aligned data |
| Any | Mockaroo | Web-based CSV/JSON synthetic data generator |

**Example: Generating synthetic user data with Python Faker:**

```python
from faker import Faker
import json

fake = Faker("en_GB")  # UK locale

def generate_test_user():
    return {
        "name": fake.name(),
        "email": fake.unique.email(),
        "phone": fake.phone_number(),
        "address": {
            "street": fake.street_address(),
            "city": fake.city(),
            "postcode": fake.postcode()
        },
        "date_of_birth": str(fake.date_of_birth(minimum_age=18, maximum_age=80))
    }

# Generate 10 test users
users = [generate_test_user() for _ in range(10)]
print(json.dumps(users, indent=2))
```

Synthetic data covers the vast majority of testing needs. Its limitation is that it does not reproduce the **edge cases and distributions** found in real production data — for example, the exotic combinations of fields that real customers produce over years of use.

**Strategy 2: Masked / Anonymized Production Data**

When **realistic volume and distributions** are required (performance testing, ML model validation, testing with real edge cases found in production), masked production data is used.

**Masking** replaces personally identifiable information (PII) with synthetic equivalents that preserve format and statistical properties:

| Original field | Masked value |
|----------------|-------------|
| `john.smith@company.com` | `user_a3f8b2@testdomain.com` |
| `John Smith` | `Alice Johnson` (random synthetic name) |
| `07700 900123` | `07700 900456` (random valid format) |
| `4111 1111 1111 1111` | `4111 1234 5678 9012` (Luhn-valid fake) |
| `NHS-1234567` | `NHS-9876543` (format-preserving) |
| Street address | Random synthetic address in the same postal region |

**Masking requirements:**
- Masking must be **one-way** (cannot be reversed to recover PII).
- Masked data must preserve **format validity** (a masked email must still look like an email; a masked phone must still be a valid format) to avoid triggering test failures due to format validation.
- Masking must be **consistent** (all references to the same real user in related tables must map to the same synthetic identity — or relationships break).
- The masking process must be **audited** and **documented** for compliance purposes (GDPR, HIPAA, CCPA).

**Key risk:** Re-identification. Even masked datasets can sometimes be reverse-engineered when combined with other data. Assess re-identification risk before using masked data in any environment.

---

**Strategy 3: Subset Production Copies**

A smaller extract of production data (with masking applied) that provides realistic data distributions for integration and performance testing without the full production volume.

**Typical use cases:**
- Integration tests that require realistic entity relationships (customer + orders + invoices + support tickets).
- Performance tests that need representative data volumes to surface realistic database query performance.
- Regression environments for complex enterprise products where synthetic data is too simplistic.

**Management requirements:**
- Refresh schedule: How often is the subset regenerated from production? (Weekly? Monthly?) Stale data accumulates state from old tests.
- Access control: Who can access this environment? (More restricted than synthetic environments.)
- Compliance documentation: Evidence that PII was masked before the subset was created.

### Test Data Management (TDM)

**Test data management** is the organizational discipline of planning, provisioning, maintaining, and governing test data across all environments and all test types.

**TDM concerns:**

**Governance:**
- Which teams own which datasets?
- Who approves the use of production-derived data in test environments?
- What is the data retention policy for test environments?

**Tooling:**
- How is synthetic data generated? (Script, factory classes, dedicated TDM tool like Informatica TDM or Delphix)
- How is data deployed to environments? (SQL seed scripts in CI? Environment provisioning pipeline?)

**Cataloguing:**
- What test personas exist in each environment? ("Gold_member_user": pre-configured Gold tier account with order history; "Locked_account_user": locked for 5 failed logins.)
- Where is this documented? (Internal wiki, README, fixture comments)

**Reset and refresh:**
- What is the baseline state? How is it restored?
- Who triggers refreshes and on what schedule?

### Determinism, Isolation, and Privacy: The Three Pillars

**1. Determinism: Tests Must Produce the Same Result Every Time**

A test that passes on Tuesday and fails on Wednesday with the same code has a **data problem**, not a code problem. Common causes:
- Shared user accounts modified by other tests.
- Time-dependent data (a "created 7 days ago" record ages to "8 days ago" and crosses a boundary).
- Cumulative state (a test creates records and does not clean them up; next run fails on duplicate key).

**Solutions:**
- Per-test setup and teardown (create data before the test, delete it after).
- Database transaction rollback: wrap each test in a transaction that rolls back — the database returns to baseline state after each test automatically.
- Ephemeral data (tests create their own data with unique identifiers to avoid collision).

**Anti-pattern: Shared mutable accounts:**
```python
# ANTI-PATTERN: Two tests share the same user; they interfere with each other
def test_user_changes_email():
    login("shared_user@example.com", "SharedPass!")
    update_email("new_email@example.com")

def test_user_checks_profile():
    login("shared_user@example.com", "SharedPass!")
    assert profile_email == "shared_user@example.com"  # FAILS because test 1 changed it
```

**Better: Each test owns its data:**
```python
@pytest.fixture
def test_user(db):
    user = create_user(email=f"test_{uuid.uuid4()}@example.com", password="ValidP@ss1")
    yield user
    delete_user(user.id)  # Teardown

def test_user_changes_email(test_user):
    login(test_user.email, "ValidP@ss1")
    update_email("new_email@example.com")
    assert profile_email == "new_email@example.com"
```

**2. Isolation: Tests Must Not Depend on Each Other**

Tests should be executable in any order and individually (not requiring a specific prior test to have run first). Dependencies between tests cause cascading failures — one failing test causes 10 others to fail.

**3. Privacy: Real PII Must Never Be in Test Environments Without Masking**

GDPR (EU), CCPA (California), HIPAA (US healthcare) and equivalent regulations impose significant obligations:
- Real customer data in an unsecured test environment is a data breach waiting to happen.
- Test environments often have lower access controls than production.
- Developer laptops with production data present are a security and compliance risk.
- Fines for PII mishandling can be severe (GDPR: up to 4% of annual global revenue).

**Rule of thumb:** Never commit real personal data to git. Never put real customer data in a test environment without documented masking approval. When in doubt, use synthetic data.

### Designing a Baseline Persona System

A **persona system** pre-creates a set of well-defined, named test users in each environment, each representing a specific account state or user type. These personas are documented and reset to their baseline state on a regular schedule.

**Example persona catalog:**

| Persona Name | Account State | Key Properties |
|-------------|--------------|---------------|
| `standard_user` | Active, basic tier | 3 past orders, no VIP status, valid payment method |
| `vip_user` | Active, Gold tier | 10 past orders, VIP status, multiple payment methods |
| `locked_user` | Locked (5 failed logins) | Account locked; awaiting admin unlock |
| `pending_user` | Pending email verification | Registered but email not verified |
| `no_payment_user` | Active, no saved payment | No credit card on file — tests payment entry flow |
| `admin_user` | Active, admin role | Full admin permissions; test admin-only features |
| `expired_subscription` | Active, subscription expired | Access to premium features revoked; trial expired |

**Documentation for each persona:**
```markdown
### vip_user
- Email: vip_user@testenv.example.com
- Password: (stored in secret manager: "test-env/vip-user-password")
- Account ID: 10042
- Tier: Gold (since 2023-01-15)
- Order history: 10 completed orders (IDs: 5001–5010)
- Payment: Visa ending 4242 (test card; never expires in test env)

**Reset procedure (run nightly):**
- Tier: Gold
- Order count: 10 (remove any orders created during testing)
- Account status: Active
- Last login: reset to 2024-01-01
```

**Reset script (nightly CI job):**
```bash
# Runs at 2 AM; restores all personas to baseline state
./scripts/reset_test_data.sh --env staging
```

This persona system allows testers to say: "I tested Story AUTH-77 using `vip_user` for the premium path and `standard_user` for the standard path." Reproducible. Documented. Clean.

## Summary

- **Synthetic data** is the safest and most maintainable default — use Faker, Factory Boy, or equivalent to generate realistic-looking, PII-free test data.
- **Masked production data** is required when realistic distributions matter — but requires rigorous masking, audit trails, and compliance sign-off.
- **Subset production copies** are used for performance and complex integration tests — with the same compliance and masking requirements as masked data.
- **Determinism**: each test owns its data and cleans up after itself; no shared mutable state.
- **Isolation**: tests can run in any order, individually, without depending on prior tests.
- **Privacy**: real PII in test environments is a compliance and security risk — always use synthetic or properly masked data.
- **Persona systems**: pre-defined, documented, regularly reset test user accounts that make test scenarios reproducible and understandable.
