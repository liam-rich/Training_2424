# Exercise: QA vs QC — Verification vs Validation

**Mode:** Conceptual (classification)  
**Time:** ~30–40 minutes  
**Relates to:** `written/2-tuesday/quality-assurance-vs-quality-control.md`, `verification-vs-validation.md`

## Instructions

1. Open `templates/scenarios.md`.
2. For **each** scenario (1–12), assign **two** labels:
   - **QA** (process-focused, preventive) **or** **QC** (product-focused, detective), and
   - **Verification** (building the product **right**) **or** **Validation** (building the **right** product),  
   **or** write **N/A** with a **one-sentence** explanation if the label does not apply cleanly.
3. **Important:** Some scenarios may be **primarily** QA **and** involve verification-like activities—choose the **best primary** label and add a **note** if dual.

## Deliverable

Create `classification_table.md` using this row format:

```text
| # | Scenario | QA or QC | V or Val | One-line rationale |
```

## Definition of Done

- [ ] All **12** scenarios classified.
- [ ] **At least four** rationales reference **specific words** from the scenario (not generic definitions).
- [ ] **At least two** scenarios flagged as **ambiguous** with your **interpretation** (disagreement is OK if argued).

## Reflection (short)

Answer in **3–5 sentences:** Why is “**QA team**” a **misleading** name for many **testing-only** groups?

## Stretch

Add **one** **new** scenario of your own (realistic software work) that is **clearly QC** but **strongly supports** future **QA** if acted on.
